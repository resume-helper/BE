## 1. 시작하기 전에

Claude에게 작업을 지시하기 전 반드시 확인할 것:

```
1. doc/TASKS.md에서 현재 Phase와 작업 상태를 확인한다
2. 관련 문서가 doc/ 내에 있는지 확인한다
3. 아래 명령 구조에 맞게 프롬프트를 작성한다
```

---

## 2. 작업 명령 구조

모든 작업 명령은 아래 형식을 따른다.

```
[컨텍스트] + [작업 지시] + [제약 조건]
```

### 형식 예시

```
# 기본형
TASKS.md의 [Phase X-Y] [작업명]을 구현해줘.
관련 문서: doc/[파일명]

# 확장형 (복잡한 작업)
TASKS.md의 [Phase X-Y] [작업명]을 구현해줘.
- 참고 문서: doc/[파일명]
- 제약: [기술 스택 / 패턴 / 금지 사항]
- 완료 기준: [완료로 볼 수 있는 조건]
```

## 3. 공용 프롬프트 패턴

팀 내에서 자주 쓰는 명령 패턴을 표준화한다.

### 3-1. 신규 기능 구현

```
TASKS.md의 Phase [번호]-[번호] "[작업명]"을 구현해줘.
doc/prd.md와 doc/ERD_DRAFT.md를 참고해서 작성하고,
완료되면 TASKS.md에서 해당 항목 상태를 🟢로 바꿔줘.
```

### 3-2. 현재 작업 상태 확인

```
TASKS.md를 읽고 현재 🟡(진행중) 또는 🔴(미시작) 항목 중
우선순위가 높은 다음 작업이 무엇인지 알려줘.
```

### 3-3. Phase 단위 작업 시작

```
TASKS.md의 Phase [번호] 작업을 시작할게.
doc/phases/[phase-폴더]/ 안의 문서를 먼저 읽고
첫 번째 🔴 항목부터 순서대로 진행해줘.
```

### 3-4. 코드 리뷰 요청

```
방금 작성한 [파일 경로] 코드를 리뷰해줘.
doc/prd.md의 요구사항을 기준으로 누락된 로직이나 개선점을 알려줘.
```

### 3-5. 버그 수정

```
[증상 설명] 문제가 발생했어.
TASKS.md에서 관련 작업 항목을 확인하고,
doc/ 문서를 참고해서 원인을 진단하고 수정해줘.
```

### 3-6. 트러블슈팅

```
[에러 메시지 또는 증상]이 발생했어.
doc/phases/[관련 phase]/TROUBLESHOOTING.md를 참고해서
원인과 해결책을 알려줘. 해결 후 해당 파일에 케이스를 추가해줘.
```

### 3-7. 작업 완료 처리

```
[작업명] 구현이 완료됐어.
TASKS.md에서 해당 항목을 🟢로 업데이트하고
다음 작업이 무엇인지 알려줘.
```

### 3-8. 설계 검토

```
[기능명]을 구현하려고 해.
doc/ERD_DRAFT.md와 doc/INFRA_DESIGN.md를 기반으로
구현 방향이 설계와 맞는지 검토하고 접근법을 제안해줘.
```

## 4. TASKS.md 운용 규칙

### 상태 아이콘

| 아이콘 | 의미 |
|--------|------|
| 🔴 | 미시작 |
| 🟡 | 진행중 |
| 🟢 | 완료 |
| ⏸ | 보류 (사유 명시) |

### 규칙

- 작업 시작 시 🔴 → 🟡 로 변경
- 작업 완료 시 🟡 → 🟢 로 변경
- Claude에게 상태 변경을 직접 시키거나, 본인이 직접 수정
- 보류 항목은 반드시 괄호 안에 사유 기재 (예: `⏸ 도메인 확보 후`)
- 담당자 컬럼은 작업 시작 전에 반드시 기입

## 5. 문서 참조 규칙

### 문서 구조

```
doc/
├── prd.md                    # 제품 요구사항 (기능 판단 기준)
├── ERD_DRAFT.md              # 데이터베이스 설계
├── INFRA_DESIGN.md           # 인프라 아키텍처
├── TASKS.md                  # 작업 현황 트래킹
├── discussion.md             # 팀 결정 사항 기록
├── API_RESPONSE.md           # API 응답 포맷, 에러코드 명세
├── CONVENTION.md             # Claude 작업 공용 프롬프트 패턴 (이 파일)
├── MODULE_STRUCTURE.md       # 멀티모듈 구조 설계
└── phases/
    ├── phase-1-foundation/   # Phase 1 상세 계획 및 문서
    ├── phase-2-core/         # Phase 2 상세 계획 및 문서
    └── phase-3-advanced/     # Phase 3 상세 계획 및 문서
```

### 문서별 사용 시점

| 문서 | 언제 참조하는가 |
|------|----------------|
| `prd.md` | 기능 구현 범위 판단, 요구사항 확인 |
| `ERD_DRAFT.md` | 엔티티 설계, 관계 확인, DB 스키마 작성 |
| `INFRA_DESIGN.md` | 인프라 관련 설정, 배포 구성 |
| `TASKS.md` | 모든 작업 시작/완료 시 |
| `discussion.md` | 팀 결정 사항이 있을 때 기록 또는 확인 |
| `phases/*/` | 해당 Phase 작업 시 세부 구현 계획 참조 |

### Claude에게 문서 위치 안내하는 법

```
# 올바른 예시
doc/ERD_DRAFT.md를 참고해서 User 엔티티를 구현해줘.

# 잘못된 예시 (위치 불명확)
ERD 보고 유저 만들어줘.
```

## 6. 코드 작업 규칙

Claude가 코드를 작성할 때 팀원 전원이 공통으로 기대하는 기준이다.

### 기술 스택

- **언어**: Kotlin
- **프레임워크**: Spring Boot 3.x
- **빌드**: Gradle (Kotlin DSL)
- **DB**: JPA + QueryDSL
- **인증**: Spring Security + JWT

### 코드 스타일

- ktlint 설정을 따른다
- 함수는 단일 책임 원칙을 지킨다
- 레이어 의존성 방향: `interfaces → application → domain` / `infrastructure → domain` (infrastructure가 domain을 구현, application은 infrastructure를 모름)
- 예외는 글로벌 핸들러(`@RestControllerAdvice`)로 처리

### 응답 형식

```kotlin
// API 응답은 공통 래퍼 사용
ApiResponse<T>
```

### 커밋 메시지

```
feat(module): [기능명] 구현
fix(module): [버그명] 수정
refactor(module): [대상] 리팩터링
docs: [문서명] 업데이트
test(module): [테스트 대상] 추가
ci: CI/CD 설정 변경
chore: 의존성·설정 변경
```

## 8. Wiki 문서화 프로세스

TASK 완료 시마다 GitHub Wiki에 기술 결정 기록을 작성한다.
이 문서는 코드 리뷰 대신 **"왜 이렇게 만들었는가"를 미래의 팀원에게 설명하는 문서**다.

### 언제 작성하는가

`doc/TASKS.md`에서 하나의 서브태스크 그룹(예: 1-3. 인증 구현)이 완료될 때마다 작성한다.
개별 항목(예: 1-3-1. 소셜 로그인)이 아닌 **서브태스크 단위**로 하나의 Wiki 문서를 작성한다.

### 파일명 규칙

```
Task-{Phase번호}-{서브번호}-{기능명}.md

예시:
  Task-1-3-인증-모듈.md
  Task-2-1-이력서-기능.md
  Task-2-2-블록-기능.md
```

### 작성 프로세스

```bash
# 1. Wiki 레포 clone (이미 있으면 pull)
git clone https://github.com/resume-helper/BE.wiki.git /tmp/BE.wiki
# 또는
cd /tmp/BE.wiki && git pull

# 2. 문서 작성 (아래 템플릿 참조)

# 3. commit & push
cd /tmp/BE.wiki
git add Task-{번호}-{기능명}.md Home.md
git commit -m "docs: Task {번호} {기능명} 기술 결정 기록 추가"
git push origin master
```

### 문서 구조 템플릿

````markdown
# Task {Phase번호}-{서브번호}. {작업명} — 기술 결정 기록

> Phase: {X-Y} | 완료일: YYYY-MM-DD | 모듈: `module-{name}`
````

## 7. 금지 사항

Claude에게 지시할 때 팀 전원이 지켜야 할 금지 규칙이다.

| 금지 사항 | 이유 |
|-----------|------|
| TASKS.md 외부에 별도 TODO 파일 생성 | 작업 현황 분산 방지 |
| doc/ 외부에 설계 문서 생성 | 문서 위치 일관성 유지 |
| 기존 ERD와 다른 테이블 구조 임의 생성 | 설계 일관성 유지 |

| 환경변수 하드코딩 | 보안 |
| .env 또는 키 파일 커밋 | 보안 |
